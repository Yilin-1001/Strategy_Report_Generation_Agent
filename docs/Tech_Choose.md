# 系统技术选型与架构设计

## 1 概述

本研究所设计的基于 RAG 的智能 Agent 系统在整体架构上遵循 **"知识层—推理层—交互层"** 三层解耦的设计原则。其中，知识层由 RAG 子系统承担，负责长期记忆的存储、检索与精排；推理层由基于 LangGraph 的多 Agent 工作流承担，负责任务编排、推理决策与人机交互；交互层由 Gradio 构建的可视化前端承担，负责生成过程的透明展示与人工审核决策。在此架构下，本文依次阐述各层的技术选型及其理论与工程依据。

## 2 RAG 子系统技术选型

RAG 子系统作为系统的"长期记忆"载体，其核心目标是面向中文企业战略文档的高精度语义检索。考虑到企业知识库具有 **文档异构性强、专业术语密集、中文语义复杂** 的特点，本研究在文档解析、文本切分、向量化、索引存储、检索与精排各环节均采取了针对性的技术选型策略。

### 表 1 RAG 子系统技术选型表

| 技术环节 | 选型方案 | 关键参数 | 选型依据 |
|---------|---------|---------|---------|
| 文档解析 | unstructured + pdfplumber + python-docx + pypdf | — | 覆盖 PDF/DOCX/TXT 多格式异构语料，pdfplumber 在表格抽取上表现突出 |
| 文本切分 | RecursiveCharacterTextSplitter（差异化策略） | 新闻 512/50；PDF 1000/200；法规 1000/150 | 依据文档类型采用不同 chunk_size 与 overlap，兼顾语义完整性与召回粒度 |
| 中文分词 | Jieba | — | 为 BM25 稀疏检索提供高质量中文 token 化支持 |
| 嵌入模型 | BAAI/BGE-M3 | 维度 1024，max_length 8192，归一化 | 多语言、多粒度、多功能统一表示，支持长文本，对中文优化优于通用模型 |
| 嵌入服务 | SiliconFlow API | 超时 30s | 相较本地推理可降低部署成本与冷启动延迟，提升系统可用性 |
| 向量数据库 | Milvus v2.6.9（Standalone 模式） | HNSW 索引，M=16，efConstruction=256 | 开源、生产级、支持 Hybrid Search；HNSW 在高维向量场景具有优异的召回-延迟平衡 |
| 度量方式 | 内积 (Inner Product) | — | 与 BGE-M3 归一化向量配合即等价于余弦相似度，计算开销更低 |
| 存储后端 | etcd + MinIO（容器化部署） | docker-compose 编排 | etcd 管理元数据强一致性，MinIO 提供 S3 兼容对象存储，工业界标准搭配 |
| 检索策略 | Hybrid Retrieval（Dense + BM25） | top_k=50，ef=128 | 稠密检索捕捉语义、稀疏检索保障关键词命中，二者互补 |
| 融合算法 | Reciprocal Rank Fusion (RRF) | 默认 RRF，可切换为加权融合 | RRF 无需归一化分数即可稳健融合异构排序，鲁棒性优于线性加权 |
| 重排模型 | BAAI/bge-reranker-v2-m3 | expansion_factor=3 | 在 top_k×3 候选中进行精排，以交互式 cross-encoder 弥补双塔模型的精度损失 |
| RAG 框架 | LangChain | — | 模块化抽象（Loader/Splitter/Retriever）成熟，社区生态完整，便于工程实施 |

在工程实现上，RAG 子系统采取 **"粗排-精排"两阶段检索范式**：第一阶段由 Milvus HNSW 稠密索引与 BM25 稀疏索引并行召回，并通过 RRF 进行无参数融合，得到 top_k×3 候选集合；第二阶段调用 BGE-Reranker-v2-m3 交互式模型对候选进行 cross-encoder 精排，最终输出 top_k 文档。该设计在召回率与精确率之间取得了较好平衡，亦是目前学术界与工业界主流的混合检索范式。值得特别说明的是，本研究针对不同文档类型（新闻、PDF 报告、DOCX 法规）采用了差异化的切分策略，避免了"一刀切"切分对长法规条文与短新闻语义边界的破坏，进一步提升了检索语义的完整性。

## 3 Agent 后端技术选型

Agent 后端是系统的"推理大脑"，承担着多智能体编排、状态管理、人机协同与质量审核四项核心职责。考虑到企业战略报告生成具有 **章节耦合性强、推理链条长、需人工审核介入** 的工作流特征，本研究采用 **状态图驱动 (State-Graph-Driven) 的多 Agent 协作架构**，并在 LLM 选型上采取"主模型—审核模型"分离策略以缓解自我评估幻觉。

### 表 2 Agent 后端技术选型表

| 技术环节 | 选型方案 | 关键参数 | 选型依据 |
|---------|---------|---------|---------|
| 编排框架 | LangGraph 0.2+ | StateGraph + 条件分支 | 相较 ReAct 框架，状态图提供显式拓扑、可中断、可回溯能力，适合长流程任务 |
| 状态持久化 | MemorySaver Checkpointer | — | 支持工作流中断、断点续跑与多轮人工审核循环 |
| 状态模式 | TypedDict + operator.add | 三层记忆架构 | 利用 Annotated reducer 实现纯累加列表，避免并发写入覆盖 |
| 主推理 LLM | DeepSeek-Chat | max_tokens 4096，timeout 180s | 国产开源模型中代码与中文写作能力突出，调用成本低，与 OpenAI SDK 兼容 |
| 审核 LLM | Kimi-K2.5（SiliconFlow） | max_tokens 8192，timeout 120s | 采用与生成模型异构的评审模型，规避同模型自评偏置，提升评审客观性 |
| API SDK | OpenAI Python SDK | — | DeepSeek 与 SiliconFlow 均兼容 OpenAI 协议，统一接口降低集成复杂度 |
| 多 Agent 角色 | Coordinator / Researcher / Analyst / Writer / Strategist / Reviewer / Archiver / ReportEvaluator | 共 8 个专职节点 | 基于战略管理"诊断—蓝图—举措"的认知流程进行角色解耦 |
| 温度策略 | Researcher 0.1 / Coordinator 0.3 / Analyst 0.5 / Writer 0.7 | 按角色梯度配置 | 检索任务低温保确定性，写作任务高温增创造性，符合任务-温度匹配原则 |
| 记忆架构 | 三层记忆模型 | 长期 (Milvus) / 工作区 (scratchpad) / 神圣上下文池 (context_pool) | 有效隔离临时草稿与已审核定稿，避免污染下游章节的上下文 |
| 人机协同 | interrupt_before 机制 | 在 human_review 节点前中断 | 实现章节级与战略蓝图级两类审核断点，支持"批准/数据/逻辑/写作"四类决策路由 |
| 工作流形态 | 两阶段战略推演 | 诊断阶段(Ch1-3) → 战略蓝图 → 举措阶段(Ch4-8) | 模拟真实战略咨询流程，蓝图作为下游章节的强约束，保障逻辑闭环 |

在架构设计上，Agent 后端的核心创新体现在两个层面：其一，采用 **三层记忆架构**（长期 RAG 记忆、章节工作区 scratchpad、跨章节神圣上下文池 context_pool）以严格区分"未定稿"与"已定稿"的信息边界，避免低质量草稿对后续推理的污染；其二，引入 **两阶段战略推演架构**——前三章为诊断阶段，第四章前由 Strategist 节点基于 SWOT 分析生成 TOWS 战略蓝图，并经人工审核后作为下游举措阶段的硬性约束注入 prepare_chapter 节点。这一设计将传统单一线性流水线升级为带反馈门控的两阶段拓扑，使得最终报告在 **战略一致性** 上具备结构性保障。此外，Reviewer 节点采用与生成模型异构的 Kimi-K2.5，从根本上规避了同模型自评带来的"自我确认偏置"，与消融实验的评审设计保持一致。

## 4 Agent 前端技术选型

前端层承担着 **生成过程透明化、人工审核高效化、最终报告可视化** 三重职责。考虑到本系统并非面向终端用户的 SaaS 产品，而是面向战略分析师的专业工作台，因此在前端选型上优先考虑 **开发效率、原生 Python 生态集成度与流式渲染能力**，而非企业级前后端分离架构。

### 表 3 Agent 前端技术选型表

| 技术环节 | 选型方案 | 关键参数 | 选型依据 |
|---------|---------|---------|---------|
| UI 框架 | Gradio (gr.Blocks) | — | 原生 Python，与 LangGraph 后端零序列化集成，开发周期较 React/Vue 方案缩短约 70% |
| 布局结构 | 三栏式工作台 | 左侧进度面板 / 中部内容区 / 右侧审核面板 | 仿专业 IDE 布局，符合"过程-内容-决策"的工作流认知顺序 |
| 流式渲染 | Generator (yield) 模式 | 节点级粒度更新 | 利用 Gradio 流式接口逐节点推送状态，实现"边生成边可视化" |
| 状态管理 | gr.State | 双状态对象 (tid + ui_state) | 维护会话级线程标识与界面级状态快照，支持跨事件上下文传递 |
| 视图切换 | 多 Group 容器隐藏切换 | 欢迎页/工作台/完成页/报告页 | 单页应用 (SPA) 范式，避免页面刷新导致的状态丢失 |
| 内容渲染 | 自定义 HTML 模板函数 | render_progress_panel / render_blueprint_review 等 | 模块化渲染函数提升可维护性，便于按面板独立迭代 |
| 视觉风格 | 自定义 CSS 主题 | theme.css (深色商务风) | 通过 CSS 变量定义统一视觉语言，呈现"高管驾驶舱"的专业气质 |
| 交互模式 | 按钮驱动的决策路由 | 批准/补充数据/重新分析/重写内容/重新生成蓝图 | 将四类修订意图显式化为按钮，降低非技术用户的认知负担 |
| 草稿库 | 历史章节下拉切换 | _get_draft_choices 动态生成 | 支持回溯查看任意已批准章节，弥补长流程任务中的上下文丢失问题 |

在前端架构上，本研究将 **生成过程的可观测性 (Observability)** 作为首要设计目标。具体而言，左侧进度面板以里程碑形式展示当前所处的章节、阶段（诊断/蓝图/举措）与流水线位置（Coord→Prep→Researcher→Analyst→Writer→Reviewer），中部工作台通过五个 Tab（检索文档、分析过程、草稿、评审详情、修改历史）将原本作为黑盒的 Agent 中间产物完整暴露给审核者，右侧面板则以语义化按钮提供四类修订决策入口。该设计有意识地避免了将 LLM 视为黑盒、仅展示最终结果的常见做法，转而将多 Agent 协作的全过程"白盒化"，这对于战略报告这类高风险、高信任要求的应用场景具有重要的实践意义。

## 5 整体技术架构

综上所述，本系统的整体技术架构可归纳为如下三层结构：

```
交互层（Gradio）  ⇄  推理层（LangGraph + 8 Agent + DeepSeek/Kimi）  ⇄  知识层（Milvus + BGE-M3 + BGE-Reranker + Hybrid RRF）
```

各层之间通过明确定义的接口契约进行解耦：知识层通过 `RAGRetriever` 暴露统一检索接口，推理层通过 `WorkflowService` 暴露工作流控制接口，交互层则通过 Gradio 事件回调与生成器协议消费上述接口。这种 **分层解耦、接口契约化** 的架构设计，不仅保障了各模块的独立可测试性（如消融实验可独立替换检索策略而不影响 Agent 编排），也为系统未来在嵌入模型升级、LLM 替换、前端重构等方面的演进预留了充分的扩展空间。
